# CONVENTIONS.md — Reca Naming and Code Conventions

Reca does not enforce these conventions — the language allows any structure.
This document describes the chosen model for the Canon Aria and Python bootstrap.

---

## Naming Prefixes

| Prefix | Meaning | Rule |
|--------|---------|------|
| `C_` | Numeric constant | Only in `constants.re` |
| `K_` | Layout/memory constant | Generated by `gen_compiler.py`. Do not edit manually. |
| `RA_` | Runtime register (volatile) | Scratch — not preserved across calls. `NEW X` (word=0). |
| `SC_` | Exire slot | Changed at runtime via Exire. `NEW X` (word=0). |
| `SYS_` | Exire number | `SYS_WRITE`, `SYS_READ`, etc. |
| `SF_` | String fragment (compiler emit) | Byte-chain Lux for IR generation. `SET X "..."` |
| `Yaku_X` | Compile rule for Aspect X | Linked graph for generating IR instruction X. |

`*_OUTER` slots (function-private RA_LINK save locations) are **no longer used**.
RA_LINK is saved/restored automatically by the call stack (`RA_SP`) on every
`Voca`/`Redi` — see `registers.re`. Non-leaf subroutines need no special
bookkeeping.

---

## NEW vs NEWREF

| Command | word | When |
|---------|------|------|
| `NEW X` | 0 | Register luces (RA_*, SC_*, scratch luces) |
| `NEWREF X` | addr(X) | Identity/relation luces — their word is used in comparisons |

**Identity luces** (must use NEWREF): `Op`, `El1`, `El2`, `Exit`, `Next`, `Entry`, `Yaku`,
`ACCORD_USE`, `LIMITER_ACCORD`, `AutoNext`, `ForType`, `SelfYaku`.

## Lumen Scanning

```re
// Scan a Data lux (lumina at offset=1):
WALK_ONE name  RA_LUX_REG  Entry     // RA_SR_OUT = exit lux where rel=Entry

// Scan an ITO lux (extra lumina at offset=ITO_SIZE=7):
WALK_ITO name  RA_ITO_REG  Op        // RA_SR_OUT = exit lux where rel=Op
```

Use `WALK_ITO` when RA_LUX holds an ITO lux with extra LINK lumina.
Use `WALK_ONE` for Data luces.

---

## AutoNext

Sequential ITO luces without NOLINK automatically receive Next → next lux.
`_link_itos(prev, nxt)` in Wave-B writes `aether[prev+SLOT_NEXT] = nxt`.

```re
ITO A  Move ...          // A.next = B
ITO B  Equal ...         // B.next = C
NOLINK                   // break — C gets no Next from B
ITO C  Jump ...          // C.next = 0
```

---

## Registries

- **LUX_REGISTRY** — all Data luces. Scanned by P0_SCAN, LIMITER, ACCORD.
- **ITO_REGISTRY** — all ITO luces. Scanned by EMIT_FUNC (compiler BFS).

An ITO lux that is also an entry point (has `LINK X Entry Yaku`) appears in **both**
registries. Use `WALK_ITO` to read its extra LINK lumina.

## Comment Syntax

### Single-line: `/`

A line starting with `/` is a comment — the entire line is ignored.

```
/This is a comment — the whole line is ignored
```

Works because `/` is not a registered command, so the loader skips the unrecognised
token to end of line.

### Inline block: `//...//`

`//` opens a comment block, the next `//` closes it. The block is **cut from the
stream** — code before and after it is read normally. Blocks may span multiple lines.

```
//cut// OK OK         ← both OKs execute
OK //cut// OK         ← both OKs execute
OK OK //cut
OK OK//               ← all four OKs execute; block spanned a line
```

The block closes at the first `//` encountered — anywhere on the line.

### What does NOT exist

- Inline `/` in the middle of a line does not open a single-line comment:
  `OK /no OK` — `/no` is treated as an unknown command (skipped), but the parser still sees it.
- `///` and longer — **not** the same as `//`. Exactly two slashes, not three or more.

### Command-like words inside prose: quote them

`check_comments.py` flags any comment line whose first word matches a real
command/macro name (`ITO`, `NEW`, `SET`, `FOR`, `LINK`, ...) — this is how
it catches a missing closing `//` swallowing real code. But documentation
prose legitimately starts sentences with these words sometimes ("Set via
Move before Exire...", or showing example syntax like "NEW MyNote"). Wrap
the word in quotes to silence the false positive without weakening the
check elsewhere:

```
/"NEW" MyNote                              ← was: NEW MyNote
/"Set" via Move before Exire; ...          ← was: Set via Move before Exire; ...
/"For" each lux with Entry→Yaku, ...       ← was: For each lux with Entry→Yaku, ...
```

The quoting is purely textual (breaks `text.split()[0] == 'NEW'`), it
doesn't change what the comment says. Every flagged line found so far has
been individually verified safe (read the surrounding `//` block, confirm
it's actually closed) before quoting — don't quote a flagged line on
sight without checking it first; the check exists to catch real bugs too.
